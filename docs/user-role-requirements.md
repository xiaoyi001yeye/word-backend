# 用户角色与权限需求文档

## 1. 文档目标

本文档用于为单词学习应用后端设计用户角色能力与权限边界，作为后续账号体系、权限控制、接口改造和数据模型设计的依据。

当前项目已经具备以下核心能力：

- 词书（Dictionary）管理
- 元单词（MetaWord）管理
- 词书与单词关联管理
- 考试（Exam）生成、提交、查看结果

当前项目尚未具备以下能力：

- 用户体系
- 登录认证
- 角色权限控制
- 班级/组织归属
- 数据隔离与资源授权

因此，本次设计目标不是直接修改代码，而是先明确角色能力、数据边界和后续实现范围。

## 2. 设计原则

### 2.1 角色清晰

系统包含三类核心角色：

- 管理员（Admin）
- 教师（Teacher）
- 学生（Student）

### 2.2 权限最小化

每个角色仅拥有完成自身任务所需的最小权限，避免学生拥有管理能力，避免教师拥有系统级配置能力。

### 2.3 数据按归属隔离

后续所有业务数据需要明确“谁创建”“归属于谁”“谁可见”“谁可操作”，避免所有人查看所有词书和考试记录。

### 2.4 先支持单角色

建议第一阶段一个用户只绑定一个主角色，不支持一个账号同时是教师和管理员。这样实现简单，权限判断更稳定。

### 2.5 与现有模块对齐

角色设计优先围绕当前已有模块展开：

- 词书
- 单词
- 考试

班级、作业、学习统计等作为角色体系的自然扩展纳入需求文档，但允许分阶段实现。

## 3. 角色定义

### 3.1 管理员

管理员负责系统级维护和全局治理，关注平台配置、用户管理、基础数据维护和全局审计。

典型职责：

- 管理用户账号
- 分配角色
- 管理教师和学生关系
- 管理系统词书和公共词库
- 查看全局统计数据
- 审核或处理异常数据

### 3.2 教师

教师负责教学组织与教学内容使用，关注班级、词书、考试和学生学习情况。

典型职责：

- 管理自己负责的班级或学生组
- 创建和维护教学词书
- 选择词书生成考试
- 查看学生考试结果和学习表现
- 给学生分配学习内容

### 3.3 学生

学生负责学习和完成考试，关注个人学习任务、词书学习和考试结果。

典型职责：

- 查看分配给自己的词书
- 参加考试
- 查看自己的考试成绩和错题
- 学习个人可见范围内的单词内容

## 4. 角色能力设计

### 4.1 管理员能力

管理员建议具备以下能力：

- 创建、编辑、禁用、启用用户账号
- 为用户分配角色：教师、学生、管理员
- 重置用户密码或触发重置流程
- 查看全部用户列表和详情
- 查看全部词书、考试、学习记录
- 导入系统词书和系统单词数据
- 删除异常词书、异常考试记录、异常关联数据
- 将词书标记为系统公共词书
- 查看系统运行统计，例如用户数、词书数、考试次数
- 配置系统级参数，例如默认考试题量、是否允许学生自由考试

建议限制：

- 管理员默认不参与教师日常教学流程
- 管理员可以查看全局数据，但不应替代教师进行常规教学操作

### 4.2 教师能力

教师建议具备以下能力：

- 查看自己负责的学生列表
- 创建、编辑、删除自己创建的教学词书
- 查看系统公共词书
- 将系统公共词书加入教学使用范围
- 为学生或班级分配词书
- 基于指定词书生成考试
- 查看自己创建的考试
- 查看所属学生的考试记录、成绩、正确率、完成时间
- 查看学生在某个词书上的学习进度
- 维护自己可管理范围内的词书单词关联

建议限制：

- 教师不能管理系统用户角色
- 教师不能删除其他教师的数据
- 教师不能修改系统级配置
- 教师默认不能直接修改系统公共词库内容，除非管理员额外授权

### 4.3 学生能力

学生建议具备以下能力：

- 查看分配给自己的词书
- 查看词书中的单词
- 发起或进入教师分配的考试
- 提交考试答案
- 查看自己的考试结果与历史记录
- 查看个人错题与薄弱词汇
- 查看个人学习统计，例如已学词数、考试次数、平均分

建议限制：

- 学生不能创建系统公共词书
- 学生不能查看其他学生成绩
- 学生不能查看教师的全量教学数据
- 学生不能删除公共数据

## 5. 权限矩阵

| 模块 | 能力 | 管理员 | 教师 | 学生 |

| 用户 | 创建用户 | 是 | 否 | 否 |
| 用户 | 修改角色 | 是 | 否 | 否 |
| 用户 | 查看本人资料 | 是 | 是 | 是 |
| 用户 | 修改本人基础信息 | 是 | 是 | 是 |
| 词书 | 查看系统公共词书 | 是 | 是 | 是 |
| 词书 | 查看自己可管理词书 | 是 | 是 | 仅已分配 |
| 词书 | 创建词书 | 是 | 是 | 可选，不建议首期开放 |
| 词书 | 编辑自己创建的词书 | 是 | 是 | 否 |
| 词书 | 删除自己创建的词书 | 是 | 是 | 否 |

| 单词 | 导入系统单词 | 是 | 否 | 否 |
| 单词 | 新增公共单词 | 是 | 可选 | 否 |

## 6. 推荐业务边界

为了让角色真正可落地，建议引入以下业务归属概念。

### 6.1 资源归属

所有核心资源建议增加归属字段：

- `createdBy`：创建人
- `ownerUserId`：资源拥有者
- `scopeType`：资源范围，例如 `SYSTEM`、`TEACHER`、`CLASS`、`PERSONAL`
- `visibleToRole` 或授权关系表：谁有权限查看

### 6.2 用户组织关系

建议引入以下组织对象：

- 班级（Classroom）
- 班级成员关系（ClassroomMember）

其中：

- 教师可以管理自己负责的班级
- 学生加入一个或多个班级
- 教师给班级分配词书和考试

如果第一阶段不做班级，也至少要有“教师-学生归属关系”，否则教师无法判断自己可以查看哪些学生数据。

### 6.3 数据可见范围

建议定义如下可见性规则：

- 管理员：可见全部数据
- 教师：可见自己创建的数据，以及自己负责学生的数据
- 学生：仅可见自己的数据，以及系统公开给自己的学习资源

## 7. 详细需求设计

### 7.1 用户基础信息

建议用户实体至少包含以下字段：

- `id`
- `username`
- `passwordHash`
- `displayName`
- `email`
- `phone`
- `role`
- `status`
- `createdAt`
- `updatedAt`
- `lastLoginAt`

建议枚举：

- `role`: `ADMIN`, `TEACHER`, `STUDENT`
- `status`: `ACTIVE`, `DISABLED`, `LOCKED`

### 7.2 认证与登录

建议后续增加登录认证能力：

- 用户名密码登录
- 登录态校验
- 未登录禁止访问受保护接口
- 不同角色访问不同接口时进行权限校验

建议首期采用：

- Spring Security
- JWT 或 Session 二选一

如果系统未来需要前后端分离和移动端支持，优先推荐 JWT。

### 7.3 词书权限规则

建议将词书拆分为三类：

- 系统词书：由管理员导入和维护，所有教师可见，学生按分配可见
- 教学词书：由教师创建，仅创建教师和其授权学生可见
- 个人词书：如果未来开放学生自建词书，则仅本人可见

建议规则：

- 管理员可管理所有词书
- 教师可管理自己创建的教学词书
- 学生只能查看分配给自己的词书

### 7.4 考试权限规则

建议将考试与发起者、参与者绑定：

- `createdByTeacherId`
- `targetStudentId` 或 `targetClassId`
- `assignedAt`
- `startedAt`
- `submittedAt`

建议规则：

- 管理员可查看全部考试
- 教师可查看自己发起的考试，以及自己学生的考试结果
- 学生只能查看自己的考试及结果

当前系统中的考试记录没有用户归属字段，因此后续实现角色功能时，这一块必须补齐。

### 7.5 学习统计需求

不同角色建议看到不同层级统计：

- 管理员：全平台统计
- 教师：班级统计、学生维度统计、词书维度统计
- 学生：个人学习统计和考试统计

建议统计指标：

- 词书数量
- 已分配词书数量
- 考试次数
- 平均分
- 最近一次考试时间
- 错题数量
- 词书完成率

## 8. 建议新增接口范围

以下为后续开发建议，不代表本次立即实现。

### 8.1 账号与认证接口

- `POST /api/auth/login`
- `POST /api/auth/logout`
- `GET /api/auth/me`
- `POST /api/users`
- `GET /api/users`
- `GET /api/users/{id}`
- `PUT /api/users/{id}`
- `PATCH /api/users/{id}/status`
- `PATCH /api/users/{id}/role`

### 8.2 教师与学生关系接口

- `POST /api/classes`
- `GET /api/classes`
- `GET /api/classes/{id}`
- `POST /api/classes/{id}/students`
- `DELETE /api/classes/{id}/students/{studentId}`
- `GET /api/teachers/me/students`

### 8.3 词书分配接口

- `POST /api/dictionaries/{id}/assign/students`
- `POST /api/dictionaries/{id}/assign/classes`
- `GET /api/students/me/dictionaries`

### 8.4 考试分配与查询接口

- `POST /api/exams/assign`
- `GET /api/teachers/me/exams`
- `GET /api/students/me/exams`
- `GET /api/students/me/exams/history`

## 9. 非功能要求

### 9.1 安全性

- 所有受保护接口必须校验登录状态
- 所有受保护接口必须校验角色权限
- 涉及资源访问的接口还必须校验数据归属
- 密码必须加密存储，不能明文保存

### 9.2 可审计性

建议记录关键操作日志：

- 用户登录
- 用户创建和禁用
- 角色变更
- 词书创建、删除、导入
- 考试创建、提交

### 9.3 可扩展性

角色模型设计上建议保留未来扩展空间，例如：

- 家长角色
- 教研组长角色
- 超级管理员角色
- 助教角色

但第一阶段不建议过度设计，先稳定三类角色。

## 10. 分阶段实施建议

### 第一阶段：账号与基础权限

目标：

- 支持用户注册或后台创建账号
- 支持登录
- 支持三类角色
- 支持接口级权限控制
- 支持用户查看本人信息

### 第二阶段：数据归属与教师学生关系

目标：

- 给词书、考试增加用户归属
- 建立教师与学生关系
- 教师可查看自己学生的数据
- 学生只能查看自己的考试和词书

### 第三阶段：班级与分配能力

目标：

- 增加班级
- 教师给班级分配词书和考试
- 学生查看被分配内容
- 教师查看班级学习统计

### 第四阶段：平台运营与增强能力

目标：

- 管理员全局统计
- 操作审计
- 错题本、学习分析、学习报告

## 11. MVP 建议

如果希望尽快上线，建议 MVP 版本只做以下能力：

- 管理员创建账号并分配角色
- 用户登录
- 基础角色权限控制
- 教师创建和管理自己的词书
- 学生查看分配给自己的词书
- 教师发起考试给指定学生
- 学生提交考试
- 教师查看学生考试结果

暂不纳入 MVP 的能力：

- 学生自建词书
- 多角色账号
- 复杂审批流程
- 细粒度自定义权限

## 12. 验收标准

当后续开始开发时，建议以以下标准验收：

### 12.1 角色正确性

- 管理员可以访问全部管理接口
- 教师无法访问管理员接口
- 学生无法访问教师和管理员接口

### 12.2 数据隔离正确性

- 学生无法查看其他学生考试记录
- 教师无法查看非自己学生的数据
- 教师无法删除其他教师创建的词书

### 12.3 核心流程可用

- 管理员可创建教师和学生账号
- 教师可给学生分配词书
- 教师可发起考试
- 学生可完成考试并查看自己的结果

## 13. 最终建议

从当前项目状态看，最合理的实现顺序不是直接加一个 `role` 字段就结束，而是一起规划以下四件事：

- 用户账号体系
- 登录认证
- 资源归属
- 角色权限控制

如果只增加角色而不增加资源归属，后续会出现“教师能看到谁的数据”“学生能访问哪些考试”的边界不清问题。

因此，建议后续开发以“管理员 / 教师 / 学生 + 数据归属 + 鉴权”作为一个完整需求一起推进。
