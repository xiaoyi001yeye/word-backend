# AI 配置管理功能需求文档

## 1. 文档目标

本文档用于定义“AI 配置管理”功能在当前 `word-backend` 项目中的业务目标、数据模型、接口范围、权限规则、页面范围与验收标准，作为后续开发、联调、自测和补充技术设计的依据。

本文档聚焦需求与落地边界，不直接修改业务代码。

## 2. 当前项目现状

结合当前主干代码，项目已经具备以下能力：

- 已有用户体系，用户表为 `users`
- 已有登录认证能力，使用 JWT 与 `/api/auth/*` 接口
- 已有用户角色枚举：`ADMIN`、`TEACHER`、`STUDENT`
- 已有当前登录用户获取能力：`CurrentUserService`
- 已有部分访问控制封装：`AccessControlService`
- 已有全局异常返回结构：`GlobalExceptionHandler`
- 已有 Flyway 迁移体系，当前最新迁移版本为 `V20`
- 已有两个前端入口：
  - `frontend`：教师 / 学生主端
  - `admin-frontend`：管理员端

当前项目尚未具备以下能力：

- 独立的 AI 配置数据模型
- AI 配置的 CRUD 接口
- API Key 加密存储能力
- 通用 AI 调用服务封装
- AI 配置管理页面

因此，本需求需要在现有用户认证、角色模型和前后端分端结构之上新增一个独立功能模块，而不是复用现有词书、学习计划或导入模块。

## 3. 功能背景

平台内用户需要调用外部 AI 服务完成短文生成、内容辅助创作、文本润色等任务。不同用户可能使用不同 AI 厂商、不同模型、不同 API 地址与不同密钥，因此系统需要提供“按用户隔离”的 AI 配置管理能力。

该能力上线后，用户可自行维护个人 AI 配置，业务模块在发起 AI 请求时只读取当前登录用户已启用的配置，避免多人共用同一套密钥或错误访问其他用户配置。

## 4. 设计目标

### 4.1 业务目标

- 支持管理员、老师、学生三类用户分别维护自己的 AI 配置
- 支持 AI 配置的新增、编辑、删除、查询
- 支持单用户保存多条 AI 配置，并选择启用项
- 支持系统按当前登录用户读取有效配置并调用 AI 服务
- 支持为后续“短文生成”“内容辅助”类功能提供统一配置入口

### 4.2 安全目标

- API Key 不得明文存储
- 各角色仅可访问自己的 AI 配置
- AI 调用时不得返回完整明文密钥
- 接口必须复用现有认证机制并进行权限拦截

### 4.3 实施目标

- 与当前 Spring Boot + PostgreSQL + Flyway 技术栈兼容
- 与当前 `CurrentUserService`、`AccessControlService` 风格一致
- 与当前前端双入口结构一致，避免把管理员入口塞回用户端

## 5. 功能范围

本期范围包括：

- AI 配置列表查询
- AI 配置详情查询
- AI 配置新增
- AI 配置编辑
- AI 配置删除
- AI 配置启用 / 禁用切换
- AI 配置基础校验
- 基于当前配置发起 AI 对话，用于验证接口是否可用
- 基于当前用户已启用 AI 配置的示例调用接口
- 前端配置管理页面
- 开发、自测、文档补充

本期不包括：

- 平台统一代充或托管第三方 API 额度
- 多人共享同一条 AI 配置
- 复杂提示词模板市场
- AI 调用账单统计、配额计费、调用限流后台
- 厂商 SDK 深度适配到每个业务细节

## 6. 用户与权限规则

### 6.1 角色定义

- 管理员：只能管理自己的 AI 配置，不可查看其他管理员、老师、学生的 AI 配置
- 老师：只能管理自己的 AI 配置
- 学生：只能管理自己的 AI 配置

### 6.2 数据隔离规则

- 每条 AI 配置必须绑定唯一 `user_id`
- 配置查询默认基于当前登录用户过滤
- 不提供“按用户 ID 查询他人 AI 配置”的管理接口
- 后端即使收到他人配置 ID，也必须校验归属后才可访问

### 6.3 接口权限规则

- 所有 AI 配置接口都要求登录
- 不使用“仅管理员可查看全部”的特权模式
- 管理接口统一使用“当前用户 + 配置归属”做授权判断

### 6.4 前端页面归属

结合当前项目的双前端结构，建议按以下方式落地：

- `frontend`：提供老师、学生的 AI 配置页面
- `admin-frontend`：提供管理员自己的 AI 配置页面

原因：

- 当前 `frontend` 在登录后若识别到 `ADMIN` 会自动跳转 `/admin/`
- 当前管理员有独立端，不宜再把管理员配置页面强塞回用户端
- 这样更符合当前项目已有结构，开发改动最小

## 7. 核心业务流程

### 7.1 配置管理流程

1. 用户登录系统
2. 进入“AI 配置管理”页面
3. 查看自己的配置列表
4. 新增或编辑 AI 配置
5. 系统校验字段合法性
6. API Key 加密后入库
7. 用户可启用或禁用某条配置

### 7.2 AI 调用流程

1. 用户在业务页面触发 AI 能力，例如生成短文
2. 后端读取当前登录用户的可用 AI 配置
3. 若指定配置 ID，则校验该配置归属于当前用户且状态为启用
4. 系统解密 API Key，拼装请求并调用第三方 AI 接口
5. 返回 AI 响应结果给业务模块

### 7.3 AI 对话测试流程

1. 用户在 AI 配置页面完成配置保存
2. 用户进入该配置的“对话测试”区域并输入一条消息
3. 前端携带配置 ID 调用 AI 对话接口
4. 后端校验该配置归属于当前用户且状态为启用
5. 系统解密 API Key，按 OpenAI-compatible Chat Completions 协议转发消息
6. 前端展示本轮用户消息与 AI 回复
7. 若成功返回内容，则可视为该 AI 接口当前可用

### 7.4 异常流程

典型异常包括：

- 用户没有任何已启用配置
- 指定配置不存在
- 配置不属于当前用户
- 对话测试时配置未启用
- 对话测试返回空响应或无法解析回复内容
- API URL 非法
- 模型名为空
- API Key 格式不合法
- 第三方 AI 接口返回认证失败、额度不足、超时或限流

## 8. 数据模型设计

### 8.1 表名建议

建议新增表：`ai_configs`

### 8.2 字段设计

| 字段名 | 类型建议 | 必填 | 说明 |
|--------|----------|------|------|
| `id` | `bigserial` | 是 | 主键 |
| `user_id` | `bigint` | 是 | 关联 `users.id` |
| `user_role` | `varchar(20)` | 是 | 冗余保存角色，便于审计 |
| `provider_name` | `varchar(100)` | 是 | AI 厂商名称，如 `OpenAI`、`ByteDance`、`Alibaba Cloud` |
| `api_base_url` | `varchar(500)` | 是 | AI 接口基础地址 |
| `api_key_encrypted` | `text` | 是 | 加密后的 API Key |
| `api_key_masked` | `varchar(50)` | 是 | 掩码展示值，如 `sk-****89ab` |
| `model_name` | `varchar(100)` | 是 | 模型名称 |
| `status` | `varchar(20)` | 是 | `ENABLED` / `DISABLED` |
| `is_default` | `boolean` | 是 | 是否默认配置，首期建议支持 |
| `remark` | `varchar(500)` | 否 | 备注 |
| `created_at` | `timestamp` | 是 | 创建时间 |
| `updated_at` | `timestamp` | 是 | 更新时间 |

### 8.3 索引与约束建议

- 主键：`id`
- 外键：`user_id -> users.id`
- 普通索引：`idx_ai_configs_user_id`
- 普通索引：`idx_ai_configs_user_id_status`
- 唯一约束建议：
  - `user_id + provider_name + model_name + api_base_url`
  - 用于避免用户重复保存同一条逻辑配置

### 8.4 状态枚举建议

- `ENABLED`
- `DISABLED`

### 8.5 是否默认配置规则

建议支持 `is_default`，原因如下：

- 当前需求明确存在“生成短文等业务场景”
- 若一个用户保存多条配置，业务模块需要确定默认使用哪一条
- 仅依赖“第一条启用配置”不稳定，后续集成复杂

建议约束：

- 同一用户最多只能有一条 `is_default = true`
- 将某条配置设为默认时，自动取消该用户其他配置的默认标记
- 默认配置必须处于启用状态

## 9. 敏感信息存储要求

### 9.1 强制要求

- `api_key` 不得明文入库
- 列表页和详情页默认不返回完整 `api_key`
- 日志中不得打印明文 `api_key`
- 异常信息中不得回显第三方密钥

### 9.2 加密方案要求

建议采用“应用层对称加密”实现，使用独立配置项管理密钥，例如：

- `security.ai-config.encryption-key`

要求：

- 不得复用 JWT `secret`
- 加密与解密逻辑封装为独立服务
- 支持后续轮换密钥的扩展

### 9.3 返回值脱敏要求

接口响应建议返回：

- `apiKeyMasked`

不返回：

- `apiKey`
- `apiKeyEncrypted`

编辑场景下，前端仅看到掩码值和“是否重新输入密钥”的状态，不回显完整明文。

## 10. 字段校验规则

### 10.1 必填项

以下字段为必填：

- `providerName`
- `apiBaseUrl`
- `apiKey`
- `modelName`
- `status`

### 10.2 格式校验

- `providerName`
  - 长度 2 到 100
- `apiBaseUrl`
  - 必须为合法 `http` 或 `https` URL
  - 不允许为空字符串
- `apiKey`
  - 长度建议 8 到 512
  - 首尾空格应自动去除
  - 不做强绑定厂商的正则校验，避免误杀不同厂商格式
- `modelName`
  - 长度建议 1 到 100
- `remark`
  - 长度不超过 500

### 10.3 业务校验

- 新增时必须提供 `apiKey`
- 编辑时若未提交新 `apiKey`，则沿用旧密钥
- 启用配置前必须保证关键字段完整
- 设为默认配置时必须同时为启用状态

### 10.4 连通性校验

建议将“保存配置”与“连通性测试”解耦：

- 保存接口只做静态字段校验
- 另提供“测试连接”或“示例调用”接口

原因：

- 第三方服务可能临时波动
- 若强绑定保存时联通校验，用户体验差，且不利于先保存后调试

## 11. 后端接口需求

以下接口均要求登录态。

### 11.1 配置列表

`GET /api/ai-configs`

功能：

- 查询当前登录用户的 AI 配置列表

支持参数：

- `status`：可选
- `providerName`：可选

返回特点：

- 仅返回当前用户数据
- 返回 `apiKeyMasked`，不返回明文密钥

### 11.2 配置详情

`GET /api/ai-configs/{id}`

功能：

- 查询当前用户某条 AI 配置详情

规则：

- 非本人配置返回 `403` 或 `404`
- 响应中不返回明文 `apiKey`

### 11.3 新增配置

`POST /api/ai-configs`

请求体建议：

```json
{
  "providerName": "OpenAI",
  "apiBaseUrl": "https://api.openai.com/v1",
  "apiKey": "sk-xxxx",
  "modelName": "gpt-4o-mini",
  "status": "ENABLED",
  "isDefault": true,
  "remark": "短文生成功能使用"
}
```

处理要求：

- `userId`、`userRole` 从当前登录用户上下文获取，不允许前端传入
- `apiKey` 入库前加密
- 返回脱敏结果

### 11.4 编辑配置

`PUT /api/ai-configs/{id}`

处理要求：

- 仅允许修改当前用户自己的配置
- 支持更新厂商、URL、模型、状态、默认标记、备注
- 若请求体不含新 `apiKey`，则保留旧值
- 若包含新 `apiKey`，则重新加密保存

### 11.5 删除配置

`DELETE /api/ai-configs/{id}`

处理要求：

- 仅允许删除当前用户自己的配置
- 若删除的是默认配置，可不自动切换新默认，交由用户后续处理
- 返回标准删除结果

### 11.6 启用 / 禁用切换

`PATCH /api/ai-configs/{id}/status`

请求体建议：

```json
{
  "status": "ENABLED"
}
```

处理要求：

- 仅允许当前用户操作自己的配置
- 若某条配置被禁用且它是默认配置，建议同步取消默认标记

### 11.7 设置默认配置

`PATCH /api/ai-configs/{id}/default`

功能：

- 将当前用户指定配置设为默认配置

处理要求：

- 仅允许设置自己的配置
- 目标配置必须为启用状态
- 自动清理同用户下其他默认配置

### 11.8 示例 AI 调用接口

`POST /api/ai/generate-reading`

用途：

- 作为业务模块接入示例，验证 AI 配置可用性

请求体建议：

```json
{
  "configId": 12,
  "topic": "environment protection",
  "wordCount": 180,
  "difficulty": "middle-school"
}
```

处理要求：

- `configId` 可选，未传时读取当前用户默认配置
- 校验配置归属与状态
- 调用成功后返回生成文本及基础元数据
- 第三方失败时返回可理解错误信息

### 11.9 连接测试接口

`POST /api/ai-configs/{id}/test`

用途：

- 测试当前用户某条配置是否可成功调用 AI 厂商接口

建议：

- 首期可选做
- 若不做独立测试接口，可由 `POST /api/ai/generate-reading` 先承担验证能力

## 12. DTO 与返回结构要求

### 12.1 列表 / 详情返回字段建议

```json
{
  "id": 12,
  "providerName": "OpenAI",
  "apiBaseUrl": "https://api.openai.com/v1",
  "apiKeyMasked": "sk-****89ab",
  "modelName": "gpt-4o-mini",
  "status": "ENABLED",
  "isDefault": true,
  "remark": "短文生成功能使用",
  "createdAt": "2026-03-30T10:00:00",
  "updatedAt": "2026-03-30T12:00:00"
}
```

### 12.2 错误返回要求

建议沿用当前项目统一异常结构，典型状态码如下：

- `400`：参数错误、URL 非法、状态非法
- `401`：未登录
- `403`：无权访问他人配置
- `404`：配置不存在
- `409`：重复配置、默认配置冲突
- `502` 或 `500`：第三方 AI 服务调用失败

## 13. 后端实现边界建议

### 13.1 包结构建议

建议新增以下包内容：

- `model`：`AiConfig`、`AiConfigStatus`
- `repository`：`AiConfigRepository`
- `dto`：创建、更新、状态切换、返回 DTO
- `service`：`AiConfigService`、`AiGatewayService`、`AiConfigCryptoService`
- `controller`：`AiConfigController`、`AiController`

### 13.2 鉴权方式建议

优先复用当前模式：

- `CurrentUserService` 获取当前用户
- 服务层按 `userId` 校验归属
- 需要时补充 `AccessControlService` 的 AI 配置权限方法

### 13.3 数据访问建议

仓库层建议提供：

- `findAllByUserId(...)`
- `findByIdAndUserId(...)`
- `existsByUserIdAnd...`
- `clearDefaultByUserId(...)`
- `findByUserIdAndIsDefaultTrue(...)`

## 14. 前端页面需求

### 14.1 页面入口

- `frontend`
  - 面向老师、学生
- `admin-frontend`
  - 面向管理员

### 14.2 页面形态

建议提供独立的“AI 配置管理”页面或弹窗，不建议散落在业务页面内部。

### 14.3 列表页要求

列表页展示：

- 模型名称
- API Key 脱敏值
- 状态
- 是否默认
- 备注

- 操作按钮：编辑、删除、启用 / 禁用、设为默认、测试

### 14.4 新增 / 编辑表单要求

表单字段：

- AI 厂商名称
- AI 接口地址
- API Key
- 模型名称
- 状态
- 是否默认
- 备注

交互要求：

- 编辑时显示脱敏后的密钥提示
- 若不修改密钥，可不重新输入

### 14.5 页面权限要求

## 15. AI 示例业务场景需求

本期至少提供 1 个业务示例接口，建议选择“生成阅读短文”，原因如下：

- 与当前英语学习产品定位匹配
- 便于快速验证配置、模型、调用链路是否可用
- 后续可扩展为作文批改、例句生成、释义润色等功能

### 15.1 示例接口入参建议

- `topic`
- `wordCount`
- `difficulty`
- `configId`（可选）

### 15.2 示例输出建议

- 生成标题
- 正文内容
- 使用的配置 ID
- 使用的厂商与模型

### 15.3 调用链路要求

- 仅使用当前登录用户自己的启用配置
- 不允许跨用户读取配置
- 保证失败信息可定位但不暴露敏感信息

## 16. 非功能要求

### 16.1 安全

- 敏感信息必须加密存储
- 认证必须复用现有登录机制
- 权限必须按用户隔离
- 禁止记录明文密钥到日志

### 16.2 性能

- 配置列表查询应为轻量接口
- 常规 CRUD 不依赖外部 AI 服务即可完成
- AI 调用接口需设置合理超时，避免长时间阻塞

### 16.3 可维护性

- 厂商调用逻辑与配置管理逻辑分层
- 配置加密逻辑独立封装
- 第三方请求参数映射尽量抽象，便于后续扩展新厂商

### 16.4 可扩展性

后续可扩展方向：

- 支持更多厂商鉴权方式
- 支持自定义请求头
- 支持系统级预设模型模板
- 支持调用日志与失败统计
- 支持业务模块按场景绑定默认配置

## 17. 数据库迁移与脚本要求

- Flyway 脚本，建议命名为 `V21__create_ai_config_tables.sql`
- 如涉及初始化枚举或约束，也应在该版本或后续版本中完成

脚本内容至少包括：

- 新建 `ai_configs` 表
- 索引
- 约束
- 默认值
- 外键

## 18. 测试要求

### 18.1 后端自测范围

- 新增配置成功
- 编辑配置成功
- 删除配置成功
- 启用 / 禁用切换成功
- 默认配置切换成功
- API Key 加密存储验证
- 非本人访问被拒绝
- 未登录访问被拒绝
- URL 非法返回 `400`
- 重复配置返回 `409`
- 示例 AI 调用成功 / 失败路径

### 18.2 前端自测范围

- 列表展示正确
- 表单新增与编辑正确
- 错误提示可读
- 编辑时密钥不回显
- 不同角色进入各自页面正常

## 19. 验收标准

满足以下条件可视为本需求完成：

- 用户可新增、编辑、删除自己的 AI 配置
- 用户无法查看或操作其他用户配置
- API Key 已加密入库，接口不回显明文
- 用户可启用 / 禁用配置，并可设置默认配置
- 系统可基于当前用户配置完成一次 AI 示例调用
- 教师 / 学生页面与管理员页面均提供对应配置入口
- 主要异常场景均有明确提示
- 完成自测与文档补充

## 20. 实施建议

建议按以下顺序开发：

1. 数据库表与 Flyway 脚本
2. 实体、仓库、DTO、服务、控制器
3. 加密服务与默认配置规则
4. 示例 AI 调用服务
5. `frontend` 页面
6. `admin-frontend` 页面
7. 联调、自测、补文档

## 21. 待确认问题

以下问题建议在正式开发前确认：

1. `apiBaseUrl` 是否允许用户填写完整路径，还是只允许填写基础域名
2. 首期是否必须支持“连接测试”独立接口
3. 示例 AI 调用接口最终是固定“生成短文”，还是抽象为通用文本生成
4. 是否需要记录 AI 调用历史
5. 是否允许学生使用 AI 功能，还是仅允许配置但暂不开放业务入口
6. 是否允许一个用户同时启用多条配置，还是只允许一条默认配置承担全部业务调用

## 22. 结论

AI 配置管理功能在当前项目中具备明确的实现基础：已有用户、角色、认证、异常处理、Flyway 与双前端结构。建议本需求采用“用户私有配置 + API Key 加密存储 + 默认配置选择 + 示例 AI 调用接口”的方案推进，实现难度可控，且能为后续 AI 业务扩展提供统一基础设施。
